The project is organized as a two-stage computer-vision pipeline in which the two modules have clearly separated responsibilities but are tightly connected by data flow and terminology. The first module, detect.py, is the main execution and orchestration layer. Its purpose is to run the complete end-to-end process over a folder of test images: load the trained YOLO detector, scan each image for traffic-light candidates, extract crops for every detected region, pass those crops into the color-classification stage, draw the resulting decision back onto the original image, and save both the intermediate crops and the final annotated outputs. In other words, detect.py is the module that controls the whole workflow and connects all individual processing steps into one usable batch-processing script.

Internally, detect.py is structured as a linear processing pipeline. The first logical part is dependency import and initialization. This part brings in file-system handling through os, image-processing operations through OpenCV, the trained YOLO model through ultralytics, and the two helper functions from classifier.py. This already shows the architectural idea of the whole project: detection and color interpretation are separated, and the top-level module is responsible for combining them.

The next logical part is model loading and input/output configuration. The YOLO model is loaded from the trained weights file, the input folder is defined, the list of files is read and sorted, and the output folders for crops and annotated detections are created if they do not already exist. This part establishes the environment in which the rest of the pipeline operates. It does not process any image content yet; instead, it prepares the detector, the input dataset, and the destinations where intermediate and final results will be stored.

After setup, the module enters its main batch-processing loop over the input files. The first responsibility inside this loop is file filtering and image loading. Only files with image extensions are accepted, which prevents accidental processing of unrelated folder contents. Each accepted image is loaded into memory, and a copy of the original frame is created for drawing. This copied frame is important because it separates analysis from visualization: the original image is used as data for cropping and classification, while the copied image becomes the canvas for rectangles and labels.

The next major logical block is detection inference. For every loaded image, the YOLO model is executed with defined inference settings. The detector returns a result object containing bounding boxes for the traffic-light candidates found in the image. This stage is the localization stage of the whole project. Its purpose is to reduce the full-image problem into a set of smaller, meaningful regions that can later be analyzed more precisely by the classifier module.

Once detections are available, detect.py enters its per-detection loop. This is the most important internal structural block in the module, because it transforms each raw detection into a fully processed result. The first part of that inner loop extracts the bounding-box coordinates and converts them to integer pixel indices. Immediately after that, the coordinates are clamped to the image boundaries. This boundary check is a defensive step whose purpose is to ensure that even if a detection box slightly exceeds the image area, cropping remains valid and safe.

The next logical part is crop extraction and crop saving. The module slices the detected region from the original frame and writes that crop into the crop-output directory using a filename that preserves both the source image identity and the detection index. This is important structurally because it exposes the intermediate data that the classification stage actually receives. These saved crops make the pipeline easier to inspect, debug, and explain, because they show exactly what visual content is passed from the detector stage into the color-analysis stage.

After the crop is prepared, detect.py performs its main cross-module handoff into classifier.py. It first calls build_color_masks(crop), which converts the crop into two evidence maps: a red mask and a green mask. It then calls predict_color(red_mask, green_mask), which compares the amount of red and green evidence and returns the final color prediction together with the raw evidence counts. This is the key integration point of the whole project. Detect.py itself does not decide what color the traffic light is; instead, it delegates that decision to the helper module and then uses the result.

The next logical part is result visualization. For each processed detection, the module draws a bounding rectangle around the detected region and places the predicted color label above it. This is the point where the abstract classification result is turned into a human-readable output. The visualization is not just cosmetic; it is the form in which the whole combined pipeline becomes directly interpretable on the original image.

Finally, after all detections in the current image have been processed, the module saves the fully annotated frame into the final output directory. This completes the per-image cycle. When the outer loop finishes, every valid input image has been turned into a final output image, and every detected region has also been archived as an intermediate crop. Structurally, detect.py therefore consists of five major layers: setup, per-image loading, per-image detection, per-detection analysis, and final output saving.

The second module, classifier.py, is the localized color-analysis stage of the project. Its purpose is much narrower and more specialized than detect.py. It does not load datasets, run object detection, manage folders, or draw outputs. Instead, it assumes that another part of the system has already isolated a candidate traffic-light crop, and it focuses only on answering one question: does the crop contain stronger red evidence or stronger green evidence? In the architecture of the whole project, classifier.py is therefore the interpretation module that operates on already localized traffic-light regions.

Internally, classifier.py is structured around two closely connected functions, each representing one stage of the color-classification logic. The first function, build_color_masks(img), is responsible for creating the intermediate visual representation used for color reasoning. Its purpose is not to make the final decision directly, but to transform the crop into two binary masks that express where red evidence and green evidence appear inside the image.

The internal structure of build_color_masks is itself composed of several logical steps. The first step is Gaussian blurring. The crop is lightly blurred to reduce small-scale noise, tiny intensity fluctuations, and compression artifacts. The purpose of this part is to make later color thresholding more stable by smoothing out random pixel-level irregularities without destroying the overall signal structure.

The next step is conversion from BGR to HSV color space. This is a central design choice of the module. HSV separates color identity from brightness and saturation more naturally than raw BGR, which makes it better suited for threshold-based color segmentation. The purpose of this conversion is to make it possible to define red and green ranges in a way that is more robust and interpretable for traffic-light analysis.

After the color-space conversion, the function builds the red mask. This logical block is especially important because red in HSV wraps around the circular hue scale. For that reason, the red evidence cannot be captured with one continuous interval. Instead, the function defines two separate red ranges, one near the lower end of the hue axis and one near the upper end, and then combines them into one red mask. The purpose of this structure is to correctly capture red pixels regardless of which side of the hue boundary they fall on.

The following logical block builds the green mask. Unlike red, green can be represented with one continuous hue range, so only one threshold interval is needed. This block marks all pixels whose hue, saturation, and value fall inside the configured green range. Its purpose is to produce the green evidence map that will later be compared directly against the red evidence map.

After both masks are created, build_color_masks performs morphological cleanup. This is another important internal stage. Raw thresholded masks often contain tiny isolated blobs, holes, or irregular fragments caused by noise, reflections, background clutter, or minor variations inside the crop. To make the evidence maps more stable, the function applies opening followed by closing to both red and green masks using a small elliptical structuring element. The purpose of opening is to remove tiny isolated foreground noise, while the purpose of closing is to fill small holes and reconnect nearby meaningful regions. This cleanup makes the masks more coherent before any numeric comparison is performed.

At the end of build_color_masks, the function returns the cleaned red and green masks. These two outputs are the full purpose of the function: they are intermediate evidence maps that encode where each target color appears in the crop. The function therefore acts as the feature-construction stage of classifier.py.

The second function in classifier.py, predict_color(red_mask, green_mask), is the decision stage built on top of those evidence maps. Its purpose is to turn the two masks into one final classification result. Internally, its structure is intentionally simple and transparent. First, it counts the non-zero pixels in the red mask and the green mask. Since the masks are binary, these counts directly represent how much red and how much green evidence has been detected inside the crop after preprocessing and cleanup.

The next logical block in predict_color is the comparison step. The function compares the two evidence totals and applies a direct dominance rule: if the red count is larger than the green count, the prediction is red; otherwise the prediction is green. This makes the function the final decision rule of the whole color-analysis stage. It does not try to model complex confidence reasoning or probabilistic behavior; instead, it converts the cleaned visual evidence into a straightforward binary decision.

The last logical part of predict_color is the return statement. The function returns not only the final textual prediction, but also the raw red and green pixel counts. This is important structurally because it preserves the numeric basis of the decision and makes the behavior easier to inspect, debug, and extend later. Even though detect.py currently uses mainly the color label for visualization, the additional returned counts expose the internal evidence behind that label.

Taken together, the two modules form a clean layered architecture. Detect.py is the outer control layer that handles the full dataset-level workflow, object localization, cross-module coordination, and output generation. Classifier.py is the inner interpretation layer that works on one crop at a time and resolves the red-versus-green decision through mask construction and evidence comparison. Their connection is very direct: detect.py isolates candidate traffic-light regions, classifier.py interprets the color state inside those regions, and detect.py then places that interpreted result back into the context of the original image. This separation of responsibilities makes the project easy to understand conceptually: one module answers where the traffic light is, and the other answers what color state it currently shows.
