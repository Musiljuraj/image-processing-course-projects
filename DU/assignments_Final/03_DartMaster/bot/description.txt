matcher.py and debug_view.py together form the “interpretation + explanation” layer of the project.

matcher.py is the module that takes the prepared live frame and the loaded template set and turns them into one structured detection result that the controller can use. It is the point in the project where the raw visual comparison actually happens. Everything before it prepares inputs for matching, and everything after it reacts to the match that this module produces. In other words, matcher.py is not responsible for frame acquisition, configuration, prediction, cooldown timing, or sending clicks. Its role is much narrower and cleaner: compare the current frame against the available templates, evaluate which template aligns best with the frame, and express that best alignment in a structured form.

The internal structure of matcher.py is intentionally compact and layered. It contains one result dataclass, one helper function, and one matcher class.

The first important part is the MatchResult dataclass. Its purpose is to define the standard match-result format used by the rest of the project. Instead of passing raw OpenCV result maps around, the module converts the detection outcome into a small structured object that contains only the information the controller actually needs. This dataclass stores the template name, the normalized score, the top-left corner of the matched rectangle, the bottom-right corner of the matched rectangle, and the center point of that rectangle. Each of these fields has a very specific purpose. The template name is used for logging and debugging so the rest of the project knows which template produced the winning result. The score expresses match strength in a unified internal convention where higher always means better. The rectangle corners describe where the winning template aligned inside the current frame. The center point is especially important because it becomes the controller’s baseline target point before any optional prediction shifts it forward.

The next part is the helper function _is_sqdiff_method(method). Its purpose is to isolate one important technical detail about OpenCV template matching: not all matching methods interpret “best result” the same way. The SQDIFF family is the key exception. For SQDIFF methods, smaller raw values are better, while for most other methods larger raw values are better. The rest of the project should not need to keep remembering this difference. So this helper function serves as a simple decision point that tells the matcher whether the current matching method belongs to that special SQDIFF family. That decision affects both where the best location is taken from and how the raw score is converted into the project’s normalized “higher is better” score convention.

The main operational part of matcher.py is the TemplateMatcher class. Its purpose is to encapsulate the actual full-frame template-matching process. The class stores the chosen OpenCV matching method when it is created so the controller does not need to pass that method into every call. This makes the matching behavior stable and tied to the lifetime of the matcher object.

Inside TemplateMatcher, the constructor __init__(method) is structurally simple. Its job is only to store the matching method as instance state. This is important because it turns the matcher into a configured object rather than a loose function call. Once created, the matcher knows which OpenCV scoring rule it is supposed to use for all subsequent comparisons.

The real work happens in match_best(frame, templates). This function is the core detection routine of the module. Its purpose is to evaluate the current prepared frame against all loaded templates and return the strongest overall match. The function’s internal logic is organized as a very deliberate sequence of logical parts.

The first logical part initializes best_result to None. This represents the fact that before any template has been checked, there is no winning candidate yet.

The second logical part loops through every template in the provided list. This establishes the module’s selection strategy: every template gets one independent full-frame evaluation, and at the end only the strongest one survives. This means the module is not trying to return multiple simultaneous candidates. It is intentionally implementing “single best match” logic.

Inside that loop, the next logical part performs cv2.matchTemplate(frame, template.image, self.method). This is the actual visual comparison step. OpenCV slides the template over the frame and produces a result map. Each location in that map represents the quality of the match if the template were placed at that top-left position. So this result map is the raw similarity surface from which the module extracts one best location.

The next logical part is cv2.minMaxLoc(result_map). Its purpose is to inspect the whole similarity surface and find the minimum and maximum values and their locations. This is necessary because depending on the matching method, either the minimum or the maximum represents the best match.

After that comes one of the most important normalization blocks in the whole module. This block checks _is_sqdiff_method(self.method). If the method belongs to the SQDIFF family, then the best location is min_loc, and the raw value is inverted into the internal score form by computing 1.0 - min_val. If the method is not SQDIFF-based, then the best location is max_loc and the score is taken directly from max_val. The purpose of this whole block is to hide method-specific OpenCV behavior behind one consistent project-wide rule: from this point onward, a bigger score always means a better match. That is what allows the controller and the rest of the project to reason about scores in one uniform way.

Once the best location for the current template has been established, the next logical part computes the rectangle geometry. The top-left location is already known from the previous step. The bottom-right corner is created by adding the template width and height. This gives the full matched rectangle inside the frame. Then the center point is computed from that rectangle. The purpose of these calculations is to translate the abstract “best location in the result map” into concrete frame coordinates that the controller can use visually and operationally.

The next logical part packages all of this information into a MatchResult object. This is the point where the module turns raw matching data into the project’s standard structured result format.

After the current template’s result has been packaged, the next logical part compares it against the current best_result. If there is no current best yet, or if the newly computed result has a higher normalized score, it replaces the existing best. This block is what enforces the “single strongest candidate wins” rule of the module.

Finally, after all templates have been processed, the function returns best_result. If the template list was empty, the result remains None. Otherwise, the returned value is the strongest match across all provided templates. This returned object then becomes the controller’s basis for click-point selection, threshold checking, debug visualization, and optional prediction.

So overall, matcher.py has a very clear responsibility and internal shape. It standardizes match results, normalizes method-specific scoring behavior, evaluates every template against the whole frame, and returns exactly one best structured detection result.

debug_view.py is the complementary explanation module. If matcher.py is the part that decides what the project “saw,” debug_view.py is the part that makes that interpretation visible. It does not participate in matching, scoring, prediction, cooldown timing, or click dispatch. Instead, it takes already-computed runtime information from the controller and renders that information into one annotated image. So while matcher.py is a decision-support module, debug_view.py is a visibility and inspection module.

Its purpose in the architecture is to turn invisible internal state into something a human can inspect visually. That makes it especially useful for debugging, validation, and understanding whether the controller’s choices are sensible. It is not producing new decisions. It is visualizing decisions that have already been made elsewhere.

The structure of debug_view.py is very simple: it contains one function, draw_match(...), but that one function is internally composed of several clearly separated logical drawing stages.

The purpose of draw_match(...) is to build one annotated debug image from the current frame and the controller’s current decision context. Its inputs reflect exactly what the controller wants to explain visually: the original BGR frame, the selected MatchResult if one exists, the configured threshold, the final click point, the recent history of center points used by optional prediction, and a flag telling whether prediction was actually used. The return value is a new annotated image, not an in-place modification of the original frame.

The first logical part of draw_match(...) creates canvas = frame_bgr.copy(). The purpose of this is to keep the original captured frame unchanged while all debug overlays are drawn onto a separate image buffer. That way the function always returns a self-contained debug image and does not risk modifying input data that might still be needed elsewhere.

The next logical part handles motion history drawing. If history_points were provided, the function iterates through them and draws each one as a small circle. For every point after the first, it also draws a line from the previous point to the current one. The purpose of this block is to visualize the recent sequence of detected centers that the controller stored for optional prediction. This makes the recent path of motion visible and helps explain why prediction may or may not have been considered trustworthy.

The next major logical part handles the “no match” situation. If match is None, then there is no matched rectangle, no detected center, and no template-specific score to visualize. Instead of failing silently, the function writes “No match result” onto the image and returns immediately. The purpose of this early return is to make the debug output still meaningful even when no target was found. It clearly communicates that the absence of a rectangle is not a rendering bug but a real detection outcome.

If a match does exist, the next logical block computes whether the match score passes the configured threshold. This is stored in the boolean passed. Based on that result, the function chooses the main annotation color: green for threshold-passing matches and orange for matches that exist but do not satisfy the action threshold. The purpose of this part is to visually mirror the controller’s acceptance rule. It makes the difference between “a match exists” and “a match is good enough to click” immediately visible.

The next logical part draws the matched rectangle and the detected center. The rectangle uses match.top_left and match.bottom_right, and the center marker uses match.center. The purpose of this block is to show where inside the frame the winning template aligned and what the baseline center point was. This is the visual translation of the structured geometric information produced earlier by matcher.py.

After that comes the logical part that draws the final click point marker, but only if click_point is not None. This is important because the final click point may or may not be identical to the detected center. In non-prediction mode, it is usually the same point. In prediction mode, it can be shifted forward. By drawing it separately as a marker, the function makes that distinction visible. This allows the debug image to explain not only where the target was detected but also where the controller actually intended to act.

The next logical section builds three explanatory text lines. The first line shows the template name and score. The second shows the detected center and whether the threshold was passed. The third shows the final click point and whether prediction was used. The purpose of these text lines is to complement the drawn graphics with precise readable runtime facts. The visual markers show geometry, while the text gives direct semantic interpretation.

The final rendering block uses cv2.putText(...) three times to place those text lines at stable positions near the top of the image. The purpose of keeping them in fixed positions is consistency: every saved debug frame can be compared visually against others without the explanatory overlay moving around.

At the end, the function returns the fully annotated image. It does not save the file itself and does not display it. That responsibility remains with the controller. So debug_view.py stays focused purely on rendering the explanation image, while the controller decides whether that image should be saved to disk.

Taken together, matcher.py and debug_view.py form a very coherent pair. matcher.py is responsible for producing the structured interpretation of the frame, and debug_view.py is responsible for making that interpretation visible. matcher.py answers “what is the strongest match and where is it?” debug_view.py answers “how can that result, together with the controller’s choice, be shown clearly on the frame?” One is the project’s compact detection engine, and the other is its compact visual explanation layer.
