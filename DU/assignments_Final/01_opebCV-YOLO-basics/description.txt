Detailed description of purpose of both modules

1. cv5_solution.py — batch image-processing module

The purpose of this module is to implement the offline, folder-based version of the project. Its job is to take a directory of still images, run YOLO object detection on every supported image file inside that directory, keep only detections belonging to one user-selected class, save cropped image regions for those detections, and also save annotated copies of the original images with bounding boxes, per-box labels, and a final per-image detection count. In the overall project, this module defines the most important core workflow and also establishes the terminology that the rest of the implementation follows: input image, selected class, detection, crop, annotated image, and output directory structure.

Internally, the module is structured as one main orchestration function supported by three small helper functions. The first helper, parse_args(), defines the full command-line interface and therefore the external control surface of the script. It collects the class filter, model selection, input and output directories, drawing color, confidence threshold, IoU threshold, line thickness, optional explicit model override, and verbose mode. Its purpose is to separate runtime configuration from processing logic, so that the rest of the module can assume all important parameters already exist in a clean and consistent form. In other words, parse_args() turns the script from a fixed hardcoded program into a reusable configurable tool.

The second helper, clamp_box(x1, y1, x2, y2, w, h), is a safety function placed between YOLO output and OpenCV image slicing. Its purpose is not to detect anything by itself, but to make the rest of the pipeline robust. Because detector coordinates may be floating-point values and can sometimes extend slightly beyond the real image boundary, this function converts them into valid integer pixel coordinates, clamps them into the image area, and ensures that the resulting rectangle is never empty. That makes cropping and drawing safe and predictable. Conceptually, this function protects the project from edge-case failures at the exact point where abstract detection geometry is converted into real image indexing.

The third helper, list_images(input_dir), is the module’s input-collection stage. Its role is to scan the user-selected directory, keep only files with supported image extensions, and return them in sorted order. This may look simple, but structurally it is important because it creates the exact ordered work queue for the whole batch run. By handling file filtering in one dedicated place, the module keeps the main processing loop cleaner and ensures that input selection is deterministic and repeatable. The shared constant IMAGE_EXTS supports this function by defining what the script considers a valid image file.

The real center of the module is main(). This function ties all other parts together into one full pipeline. Its first logical block loads the command-line arguments and validates that the input directory really exists and is a directory. This is an early validation stage whose purpose is to fail fast before model loading or processing begins. The next logical block prepares the output directory structure. Rather than modifying source files, the module creates a base output folder and inside it two specialized subdirectories: crops for extracted object cutouts and annotated for derived images with visual overlays. This separation is one of the key design choices of the module, because it keeps original inputs untouched and keeps the two result types clearly organized.

The next logical block inside main() performs model selection and YOLO initialization. The purpose of this part is to resolve whether the script should use the default yolov8{size}.pt naming scheme or an explicitly supplied model reference. Once the model is created, the script also retrieves the class-name mapping if it exists. This is followed by the input discovery phase, where the module calls list_images() and stops if no valid images are found. At that point, the script has everything it needs: configuration, verified inputs, prepared outputs, and an initialized detector.

Another important logical part inside main() is the setup of run-local semantic values: the chosen BGR drawing color, the selected numeric class_id, and the internal helper class_name_for(cid). The purpose of class_name_for() is to translate the selected class ID into a readable label when possible, while still remaining robust if the model does not expose names in the expected format. This small helper exists so that later annotation code can stay simple and consistently refer to one readable class label.

The main processing loop over files is the core execution block of the whole module. For each image path, the script first tries to load the image using OpenCV. If loading fails, the module prints a warning and continues, which means the batch run is designed to be resilient to isolated bad files instead of aborting everything immediately. Once an image is loaded, the script stores its dimensions, creates an annotated copy, and runs YOLO inference directly on the image array. One of the most important design decisions here is that class filtering happens already at inference time through classes=[class_id]. The purpose of that choice is to keep the entire downstream logic aligned with one selected semantic category and avoid unnecessary post-filtering of unwanted detections.

After inference, another logical block processes all detections in the current image. This block iterates through YOLO’s returned boxes, increments the per-image detection count, clamps coordinates through clamp_box(), extracts the crop, computes confidence, builds an informative crop filename, and saves the crop into crops_dir. The purpose of this block is twofold: first, convert each accepted detection into a reusable extracted object image; second, build a traceable naming scheme that encodes origin image, detection index, selected class, and approximate confidence into the filename. That makes the saved crops understandable even outside the script.

The next logical part of the same loop handles visual annotation. For every detection, the script draws a bounding box on the annotated copy and adds a short per-box label containing the class name and confidence value. The purpose here is to create a human-readable interpretation layer on top of the raw image, so that the user can immediately verify what was detected and how strongly the model believed in that detection. The code also adjusts the text position depending on box location so labels stay visible near the top edge.

Once all detections for an image are processed, the module enters the final per-image finishing block. It overlays the total number of detections for the selected class in the lower part of the image, writes the annotated image into ann_dir, and prints a short console progress message with the image name and detection count. The purpose of this finalization stage is to produce one complete summary artifact for each input image and to give the user lightweight monitoring feedback during the batch run. After the loop ends, main() prints final output paths, which closes the pipeline with a clear summary of where results were stored.

In structural terms, cv5_solution.py is therefore a classic staged offline pipeline: configuration -> validation -> output preparation -> model initialization -> image collection -> per-image detection and export -> completion summary. Its design is linear, deterministic, and oriented around processing a finite set of independent source images.

2. cv5_videostream.py — live-stream processing module

The purpose of this module is to implement the real-time variant of the same detection idea. Instead of reading a folder of still images, it reads a continuous live stream or video source, runs YOLO detection on each incoming frame, draws bounding boxes and text overlays in the same general style as the batch module, shows the annotated result in an OpenCV window unless running headless, and can optionally save annotated video output and sampled crops. In the overall project, this module reuses the same semantic core as the offline script, but it adapts the outer workflow from a finite image collection to an ongoing frame loop. That makes it the streaming counterpart to the batch module.

Its internal structure is again centered around main(), but it uses helper functions that support the needs of streaming. The first helper, parse_args(), defines the module’s runtime interface. It preserves the shared controls from the batch module such as selected class, model size or explicit model, annotation color, confidence and IoU thresholds, line thickness, and verbose mode. On top of that, it introduces stream-specific controls: the source URL or video path, window title, optional FPS overlay, headless mode, optional annotated video output file, optional crop output directory, frame-based crop sampling interval, and a maximum number of crops per frame. The purpose of this function is to make the live pipeline flexible without changing its core logic. It is the place where the module exposes the difference between offline and real-time operation.

The function clamp_box() plays the same structural role here as in the batch module. Its purpose remains to clean detector coordinates before they are reused for cropping and drawing. This consistency matters because it means the project treats detection rectangles in the same robust way regardless of whether they come from a still image or a video frame. The function class_name_for(names, cid) also exists for the same conceptual reason as its batch counterpart: it translates numeric class IDs into readable labels when the model exposes such metadata and falls back to the numeric form otherwise. In the stream module, it is defined at module level because it serves as a stable general-purpose utility for frame annotation.

The heart of cv5_videostream.py is again main(), but its internal logic is shaped around a continuous loop rather than a finite directory traversal. The opening logical block parses arguments, resolves the model name, initializes YOLO, retrieves class metadata, normalizes the selected class and box color, and determines the source to read from. The default source is an MJPEG URL intended for WSL usage, which is a very important design detail: the module is built around the practical assumption that direct webcam capture inside WSL is often not the right approach. In other words, this part of main() establishes not only the detector but also the environment-specific acquisition method.

The next logical block opens the capture source with cv.VideoCapture(source) and immediately checks whether it opened successfully. If not, the module terminates with a detailed WSL-oriented troubleshooting message. The purpose of this block is to fail with meaningful diagnostics at the acquisition layer, because in a streaming application the source connection is just as critical as the detection model. Verbose startup logging follows, summarizing model choice, source, class, thresholds, colors, and optional saving configuration. This gives the live script a clear startup state before frame processing begins.

Another setup block prepares optional outputs and runtime state. The writer is initialized as None because a VideoWriter cannot be configured properly until a real frame has been received and its dimensions are known. The crop output directory is created only if requested. The variables last_t and fps establish the timing state for optional FPS display, and frame_idx tracks how many frames have been processed. If the script is not headless, it also creates the named display window. The purpose of this entire setup section is to prepare all long-lived stream state before entering the main processing loop.

The continuous while True loop is the central logical structure of the whole live module. Each iteration represents one full processing cycle for one incoming frame. The first sub-block attempts to read a frame from the capture source. If the read fails or no frame is returned, the stream is considered exhausted or broken, and the loop exits. This gives the live pipeline a clean termination condition that is driven directly by input availability. Once a frame is available, the code increments the frame counter, records height and width, and creates an annotated copy that will carry all visualization overlays.

The next sub-block runs YOLO inference on the current frame, again using classes=[class_id] so that only the selected semantic category is processed. This mirrors the core design of the batch module and keeps the project conceptually unified: both modules always organize all downstream work around one chosen class. After inference, the script extracts the first result object, accesses its boxes, and initializes a per-frame detection counter. The purpose of this stage is to convert one live frame into a structured set of class-filtered detections ready for annotation and optional export.

The detection-processing block then iterates over all boxes in the current frame. For each box, the script increments the frame-local detection count, clamps coordinates, reads the confidence score, draws the bounding rectangle, and draws the per-box label consisting of class name and confidence. Structurally, this is the live equivalent of the corresponding per-image processing block in the batch module. The same detection semantics are preserved; only the time context changes from “current image in a folder” to “current frame in a stream.”

Inside that same detection loop is an optional crop-saving block that is more carefully controlled than in the offline module. Because a stream can produce huge numbers of detections over time, the module saves crops only when crop saving is enabled, only on frames whose index matches the specified sampling interval, and only up to the per-frame crop limit. The purpose of this design is to preserve the useful idea of extracted object crops without allowing live processing to explode into unmanageable file counts. This is one of the clearest examples of how the stream module keeps the same project semantics while adapting them to real-time constraints.

After all detections for the frame are handled, the module enters its frame-finalization stage. It draws the total number of detections of the selected class near the bottom of the frame, optionally computes and overlays FPS in the top-left area, lazily initializes the VideoWriter if needed and if a save path was requested, writes the annotated frame to the output video when appropriate, and then either displays the frame in a window or continues silently in headless mode. The quit logic is also part of this stage: in windowed mode, pressing ESC or q exits the loop. The purpose of this block is to produce the complete visible and/or saved result for one frame before moving on to the next one.

Once the stream loop finishes, the module performs a cleanup block that releases the capture object, releases the writer if one exists, destroys windows if GUI mode was used, and optionally prints a final completion message in verbose mode. This ensures the real-time pipeline closes its resources properly and leaves the environment in a clean state.

In structural terms, cv5_videostream.py is therefore a stateful real-time pipeline: configuration -> model and source setup -> optional output setup -> repeated frame acquisition and processing -> live visualization and/or saving -> cleanup. Unlike the linear finite structure of the batch module, this module is built around a persistent loop and ongoing runtime state, which is exactly what makes it suitable for live-stream processing.

Relationship between the two modules

Together, the two modules form one coherent project with two execution modes built on the same detection idea. cv5_solution.py is the offline, deterministic, folder-based implementation intended for processing a closed set of input images and producing a stable collection of crops and annotated outputs. cv5_videostream.py is the real-time, frame-based implementation intended for continuously reading a source, visualizing detections live, and optionally saving derived outputs over time. Their shared terminology, class-filtering strategy, coordinate-safety logic, and annotation style make them conceptually consistent, while their outer control flow differs according to whether the input is finite and static or continuous and live.
