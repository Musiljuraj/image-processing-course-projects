DETAILED DESCRIPTION OF THE PURPOSE, INTERNAL STRUCTURE, FUNCTIONS, AND LOGICAL PARTS
OF THE WHOLE PROJECT

======================================================================
0) OVERALL PROJECT ARCHITECTURE
======================================================================

This project is a complete eye-state recognition pipeline built around two
connected ideas:

1. A classical computer-vision localization branch:
   - detect faces in video frames,
   - select one main face,
   - detect eyes and mouth inside that face.

2. A runtime eye-state classification branch:
   - originally heuristic, based on thresholding / blob analysis / circles,
   - later replaced by an LBP-based learned classifier,
   - while still keeping the heuristic classifier as a compatible fallback.

The project also includes a full training and experiment-search side:
- load the mrlEyes training dataset,
- preprocess eye images,
- compute LBP feature descriptors,
- train a classifier,
- compare multiple configurations,
- choose the best configuration,
- run a final full video pass with that chosen setup,
- evaluate against ground truth,
- save reports.

So the full project can be understood as a layered pipeline:

raw training dataset
    -> structured training records
    -> shared preprocessing
    -> LBP feature extraction
    -> classifier training
    -> reusable runtime model bundle

real input video
    -> frame-by-frame face detection
    -> one selected main face
    -> eye localization
    -> frame-level eye-state classification
    -> stored frame results
    -> evaluation against ground truth

And above that, there is one orchestration layer:

experiment search
    -> rank results
    -> select best configuration
    -> run final pipeline
    -> save final summary outputs

The modules are arranged so each one owns one clear responsibility. This is the
main structural strength of the project. The code is not one large monolithic
script. Instead, it is split into modules that correspond to natural logical
stages of the system.

The modules are:

1. auto_select_and_run.py
2. experiment_search.py
3. eye_training_io.py
4. eye_preprocessing.py
5. eye_lbp_dataset.py
6. lbp_features.py
7. eye_lbp_classifier.py
8. eye_state_lbp.py
9. eye_state.py
10. detectors.py
11. main.py
12. evaluation.py

What follows is a detailed explanation of each of them.

======================================================================
1) MODULE: auto_select_and_run.py
======================================================================

OVERALL PURPOSE

This module is the top-level automatic orchestration layer of the whole final
solution.

It does not implement any low-level computer-vision or machine-learning logic
itself. Instead, it coordinates already existing lower layers into one fully
automatic workflow:

experiment search
    -> best-result selection
    -> conversion of the winning experiment into runtime configs
    -> one final full rerun using those configs
    -> saving of final summary files

So its real purpose is:
“Turn the whole project into one push-button final solution.”

It is the module that makes the project feel complete, because it connects:
- comparison of configurations,
- automatic model/setting selection,
- final execution of the best solution.

----------------------------------------------------------------------
INTERNAL STRUCTURE
----------------------------------------------------------------------

The module is small and very focused. Its structure is:

A. imports
B. automatic final-run configuration constants
C. output helper functions
D. full automatic orchestration function
E. thin main() entry point

This structure is intentionally minimal because the module is supposed to be
only the final controller, not another algorithm-heavy layer.

----------------------------------------------------------------------
A. IMPORTS
----------------------------------------------------------------------

The module imports:
- Path for output-path handling,
- several helpers from experiment_search.py,
- run_final_pipeline from main.py.

This already reveals the logic of the module:
- experiment_search.py handles comparison and ranking,
- main.py handles actual execution of the final runtime pipeline,
- auto_select_and_run.py only connects the two.

----------------------------------------------------------------------
B. AUTOMATIC FINAL-RUN CONFIGURATION
----------------------------------------------------------------------

The constants at the top define how the top-level automatic workflow behaves.

Their purpose is to centralize policy decisions such as:
- how many frames should be used in the search,
- how many training records per class should be used in experiment search,
- whether preview should be shown,
- how the final automatic run should be named,
- what filename prefix should be used for outputs.

This part exists so the orchestration logic below stays readable.
Instead of hard-coded literals inside the workflow, the workflow reads clearly
from named constants.

----------------------------------------------------------------------
C. OUTPUT HELPER FUNCTIONS
----------------------------------------------------------------------

1. _get_auto_selection_output_paths(paths)

Purpose:
- extend the standard experiment-search output-path set with one additional
  path used only by the top-level orchestration layer.

Why it exists:
- experiment_search.py already knows the standard experiment report paths,
- but auto_select_and_run.py needs one extra summary file that links:
  - the winning experiment,
  - and the final rerun produced from that winner.

So this function is a small adapter between:
- experiment-level output naming,
- and top-level orchestration output naming.

2. _save_auto_selection_summary(best_experiment_result, final_run_result, summary_path)

Purpose:
- create one plain-text summary that explicitly connects:
  - the best experiment found during search,
  - the actual full final rerun executed using that chosen configuration.

Why it exists:
- without this helper, the user would have to inspect multiple output files
  separately to understand the whole story,
- with it, there is one human-readable bridge between:
  - search phase,
  - and final execution phase.

Logical structure inside it:
- extract the final evaluation summary from the final run,
- extract the accuracy/timing substructures,
- build a list of text lines,
- create the output directory if necessary,
- save the summary file.

This is a reporting helper, not a computational helper.

----------------------------------------------------------------------
D. FULL AUTOMATIC ORCHESTRATION
----------------------------------------------------------------------

3. auto_select_best_and_run_final()

This is the core function of the module.

Purpose:
- execute the entire final automatic workflow from experiment search to final
  rerun.

Its internal steps are:

Step 1:
Run experiment search with the configured search settings.

Step 2:
Build output paths for the orchestration layer.

Step 3:
Save the full experiment-search report.

Step 4:
Extract the best experiment result.

Step 5:
Save a short “best experiment” summary.

Step 6:
Convert the best experiment result into the exact runtime configs required by
main.py.

Step 7:
Run the full final pipeline with those chosen configs.

Step 8:
Save one top-level automatic-selection summary that links:
- winner from search,
- final rerun output.

Step 9:
Return one structured orchestration_result dictionary containing all major
outputs from every stage.

Why this function exists:
- it is the final project-level workflow,
- it makes all lower modules operate as one finished system,
- it also preserves all important intermediate outputs rather than only
  printing a final result.

The returned dictionary is important because it keeps the function reusable for:
- direct script execution,
- debugging,
- later extension,
- tests.

----------------------------------------------------------------------
E. MAIN ENTRY POINT
----------------------------------------------------------------------

4. main()

Purpose:
- provide the default script entry point for the final automatic solution.

Internal logic:
- call auto_select_best_and_run_final(),
- extract the most important top-level results,
- print a compact console summary:
  - selected experiment index,
  - selected accuracy,
  - selected configs,
  - saved output paths.

Why it exists:
- gives the project a clean final “run this file” entry point,
- produces a small final checkpoint for the user.

This function does not do any real orchestration work itself. It only delegates
to the main orchestration function and prints the final summary.

======================================================================
2) MODULE: experiment_search.py
======================================================================

OVERALL PURPOSE

This module is the experiment-comparison layer of the project.

Its purpose is to systematically test multiple preprocessing / LBP /
classifier combinations on the real video pipeline, evaluate them against the
ground truth, rank them, and expose the best result in a form reusable by the
final orchestration layer.

This module exists because by the time the project reached its final stage,
the building blocks already existed:
- training-data loading,
- preprocessing,
- LBP feature extraction,
- classifier training,
- runtime video pipeline,
- evaluation.

What was still missing was:
- a controlled way to compare configurations,
- a ranking policy,
- a way to identify the winning setup and reuse it.

That is exactly what this module does.

----------------------------------------------------------------------
INTERNAL STRUCTURE
----------------------------------------------------------------------

The module is large, but its structure is clean and layered:

A. imports and reused lower-level dependencies
B. runtime constants reused from the final pipeline
C. project path helpers
D. frame-storage helper
E. configuration helpers
F. training-record loading helpers
G. one-model build helper for experiment runs
H. one-experiment runner
I. ranking / formatting / report helpers
J. full search orchestration
K. standalone quick-run entry point

This structure mirrors the natural life cycle of experiment search.

----------------------------------------------------------------------
A. IMPORTS
----------------------------------------------------------------------

The module imports:
- copy / product / Path / timing / gc,
- OpenCV and NumPy,
- localization helpers from detectors.py,
- training-data helpers from eye_training_io.py,
- config validators from preprocessing / LBP / classifier modules,
- matrix-building helpers from eye_lbp_dataset.py,
- classifier training helper,
- runtime LBP frame classifier,
- evaluation helpers.

This import set shows that experiment_search.py is a connector module.
It is not a low-level algorithm module. It pulls together many lower layers.

----------------------------------------------------------------------
B. RUNTIME CONFIGURATION REUSED FROM MAIN PIPELINE
----------------------------------------------------------------------

These constants mirror the behavior of the final pipeline:
- mouth detection enabled,
- face-detection downscaling,
- LBP fallback policy,
- preview settings,
- quick-test defaults.

Why this part matters:
- the experiment phase should evaluate something close to the real final
  runtime behavior,
- otherwise the “best” experiment would not necessarily be best for the actual
  final run.

So these constants keep search-mode runtime aligned with final runtime.

  - full experiment search report,

Why it exists:

----------------------------------------------------------------------
D. CONFIGURATION HELPERS
----------------------------------------------------------------------

7. ensure_config_list(configs, config_name)

Purpose:
- validate that an incoming config collection is:
  - a list,
  - non-empty,
  - containing only dictionaries.

Why it exists:
- experiment generation depends on Cartesian product across clean config lists.

8. get_recommended_experiment_configuration_lists()

Purpose:
- define the built-in default experiment set.

Design:
- 1 preprocessing config,
- 3 LBP configs,
- 1 classifier config.

Why this exact design:
- keeps the experiment space small,
- satisfies the assignment idea of comparing multiple LBP configurations,
- avoids a combinatorial explosion.

9. build_experiment_configurations(...)

Purpose:
- build the full Cartesian product of all supplied config lists.

Return:
- explicit per-experiment configuration dictionaries containing:
  - experiment_index,
  - preprocessing_config,
  - lbp_config,
  - classifier_config.

Why it exists:
- converts independent config axes into concrete runnable experiment bundles.

----------------------------------------------------------------------
E. TRAINING-RECORD LOADING HELPERS
----------------------------------------------------------------------

10. _validate_max_training_records_per_class(...)

Purpose:
- validate the optional balanced-subset limit.

This allows search mode to run on:
- full dataset,
- or limited balanced subsets for quicker tests.

11. load_training_records_for_experiments(...)

Purpose:
- load training records in a way suitable for experiment search.

Two modes:
- full-dataset mode,
- balanced-subset mode.

Why the subset mode exists:
- repeated experiments are expensive,
- loading a balanced subset directly from disk reduces memory use and speeds
  up search.

Internal logic in subset mode:
- collect image paths,
- parse filenames,
- keep separate counts for closed/open classes,
- build records only until both classes reach the requested cap,
- enforce presence of both classes.

This function is important because it reduces repeated overhead during
multi-configuration search.

----------------------------------------------------------------------
F. MODEL-BUILDING HELPER FOR ONE EXPERIMENT
----------------------------------------------------------------------

12. build_eye_lbp_model_from_training_records(...)

Purpose:
- build one trained model bundle from already loaded training records.

Why it exists:
- experiment search should not reload the dataset from disk for every
  experiment,
- instead, one shared pool of training records is reused,
- only preprocessing / LBP / classifier training are repeated per config.

Internal stages:
- validate configs,
- prepare training feature records,
- build X_train, y_train, metadata,
- train classifier,
- compute model/training statistics,
- return a lean model bundle,
- explicitly delete bulky intermediates.

Important design difference from the full startup model bundle:
- this experiment version does not keep all bulky training structures alive,
- that keeps experiment memory usage lower.

----------------------------------------------------------------------
G. ONE-EXPERIMENT RUNNER
----------------------------------------------------------------------

13. run_one_experiment(...)

Purpose:
- execute one full end-to-end experiment on the real video.

This is the core experimental unit of the module.

Its internal flow is:

1. validate inputs,
2. build one model for the current config,
3. open the video,
4. initialize per-run state,
5. optionally create preview window,
6. loop over frames:
   - grayscale conversion,
   - face detection,
   - main-face selection,
   - face-parts detection,
   - LBP frame classification,
   - timing measurements,
   - result storage,
   - temporal-state updates,
   - optional preview handling,
7. release resources,
8. evaluate frame results against ground truth,
9. build one structured experiment_result dictionary,
10. clean up memory,
11. return the experiment result.

Why this function matters:
- it is the bridge between:
  - configuration definition,
  - and measurable experiment outcome.

This is where one abstract experiment configuration becomes actual accuracy,
timing, and frame-level prediction results.

----------------------------------------------------------------------
H. RANKING / FORMATTING / REPORT HELPERS
----------------------------------------------------------------------

14. rank_experiment_results(experiment_results)

Purpose:
- sort finished experiments from best to worst.

Ranking policy:
1. higher accuracy,
2. lower total frame mean time,
3. lower classification mean time,
4. lower model-build time.

Why this exists:
- makes the “best” result explicit and deterministic.

15. get_best_experiment_result(search_result)

Purpose:
- return the best-ranked experiment result from a completed search bundle.

Why it exists:
- provides a clear semantic helper instead of forcing callers to read
  ranked_results[0] directly.

16. extract_runtime_configs_from_experiment_result(experiment_result)

Purpose:
- turn one experiment result back into the exact three runtime config
  dictionaries needed by main.py.

This is the key handoff from:
- search result
to
- final runtime execution.

17. format_one_experiment_result(experiment_result, rank_index=None)

Purpose:
- convert one experiment result into readable multiline text.

It includes:
- outcome quality,
- timing,
- training/model statistics,
- configs,
- full evaluation summary.

18. format_experiment_search_summary(search_result)

Purpose:
- format the whole ranked search result into one report text.

19. format_best_experiment_summary(experiment_result)

Purpose:
- create a shorter winner-only summary.

20. save_experiment_search_report(search_result, report_path)

Purpose:
- save the full formatted ranked report.

21. save_best_experiment_report(experiment_result, report_path)

Purpose:
- save the shorter best-only summary.

Together, these formatting/report helpers form the text-reporting interface of
the experiment layer.

----------------------------------------------------------------------
I. FULL SEARCH ORCHESTRATION
----------------------------------------------------------------------

22. run_experiment_search(...)

Purpose:
- run the full multi-configuration experiment search.

This is the main public orchestration function of the module.

Its internal logic is:

1. build paths and output directories,
2. validate subset limit,
3. fill in missing config lists from recommended defaults,
4. build explicit experiment configurations,
5. load training records once,
6. load detector cascades once,
7. run run_one_experiment(...) for each configuration,
8. attach experiment indices,
9. rank results,
10. return one structured search_result bundle.

Why it matters:
- it turns the whole experiment-search logic into one callable workflow,
- it keeps expensive shared inputs reused across experiments,
- it is the function called by auto_select_and_run.py.

----------------------------------------------------------------------
J. STANDALONE QUICK TEST MODE
----------------------------------------------------------------------

23. main()

Purpose:
- run a default quick experiment search.

It uses:
- recommended 3 experiments,
- limited training subset,
- limited frame count,
- no preview.

Why this exists:
- convenient smoke-test mode,
- allows quick manual inspection of search behavior without launching a full
  final automatic workflow.

4) MODULE: eye_preprocessing.py
======================================================================

OVERALL PURPOSE

This module is the shared eye-image normalization layer.

Its purpose is to take raw eye images or runtime eye ROIs and convert them into
a stable standardized image representation before LBP feature extraction.

It is shared by:
- training-side images,
- runtime eye ROIs.

That shared use is critical because the classifier should see the same type of
input during training and during runtime inference.

The preprocessing pipeline is:

raw image
    -> grayscale conversion
    -> fixed-size resize
    -> optional central-band crop
    -> optional contrast normalization
    -> optional light filtering

----------------------------------------------------------------------
INTERNAL STRUCTURE
----------------------------------------------------------------------

A. default preprocessing constants
B. low-level validation helpers
C. public configuration helpers
D. image-level preprocessing helpers
E. end-to-end single-image preprocessing
F. record-level wrappers
G. preprocessing summary helpers

----------------------------------------------------------------------
A. DEFAULT CONFIGURATION CONSTANTS
----------------------------------------------------------------------

These constants define:
- default target size,
- default crop band,
- default contrast method and CLAHE settings,
- default filter name and kernel sizes.

Why they exist:
- centralize preprocessing policy,
- keep later logic readable,
- keep training/runtime preprocessing aligned.

----------------------------------------------------------------------
B. VALIDATION HELPERS
----------------------------------------------------------------------

1. _is_grayscale_image(image)
- checks whether the image is already single-channel grayscale.

2. _validate_target_size(target_size)
- validates (width, height) size tuples.

3. _validate_ratio(value, name)
- validates ratio values in [0, 1].

4. _validate_analysis_band(top_ratio, bottom_ratio)
- validates the crop interval used for the vertical analysis band.

5. _validate_odd_kernel_size(kernel_size, name)
- validates odd 2D kernel sizes.

6. _validate_odd_integer(value, name)
- validates positive odd scalar kernel size.

These helpers exist so later preprocessing code can assume valid inputs.

----------------------------------------------------------------------
C. PUBLIC CONFIGURATION HELPERS
----------------------------------------------------------------------

7. get_default_preprocessing_config()

Purpose:
- return a fresh default preprocessing config dictionary.

8. validate_preprocessing_config(preprocessing_config=None)

Purpose:
- merge caller config with defaults,
- validate and normalize the complete config.

Why it matters:
- all later preprocessing functions rely on this stable normalized config
  structure.

----------------------------------------------------------------------
D. IMAGE-LEVEL HELPERS
----------------------------------------------------------------------

9. convert_eye_to_grayscale(image)
- supports grayscale, BGR, and BGRA input,
- converts everything to grayscale.

10. resize_eye(gray_image, target_size)
- resize one grayscale eye image to the standard fixed size.

11. crop_eye_analysis_band(gray_image, analysis_top_ratio, analysis_bottom_ratio)
- keep only the central vertical band of the eye.

12. normalize_eye_contrast(...)
- apply:
  - none,
  - histogram equalization,

13. filter_eye_image(...)
- apply:
  - none,
  - Gaussian blur,

Together these functions are the building blocks of the shared preprocessing

6) MODULE: lbp_features.py

OVERALL PURPOSE

This module is the descriptor engine of the project.

Its job is to convert one already preprocessed grayscale eye image into an LBP

- record enrichment with LBP outputs.

This is the exact point where image representation becomes numeric texture
representation.

Purpose:

This is the main high-level descriptor function of the module.

8) MODULE: eye_state_lbp.py
======================================================================

OVERALL PURPOSE

This module is the runtime LBP eye-state decision layer.

It sits on top of:
- eye ROI extraction,
- runtime preprocessing,
- LBP feature extraction,
- trained classifier inference.

Its job is to:
- classify each detected eye separately,
- convert classifier outputs into one unified open-score,
- aggregate multiple eye predictions into one frame-level label,
- optionally fall back to the older heuristic classifier.

This module is the practical learned replacement for the heuristic runtime
classifier.

----------------------------------------------------------------------
INTERNAL STRUCTURE
----------------------------------------------------------------------

A. runtime constants
B. label/model normalization helpers
C. failed-eye record builder
D. single-eye LBP classification
E. eye-prediction aggregation
F. public frame-level interface

----------------------------------------------------------------------
A. RUNTIME CONSTANTS
----------------------------------------------------------------------

LBP_FRAME_OPEN_THRESHOLD
- used to break ties through mean open-score.

LBP_STRONG_OPEN_SCORE_THRESHOLD
- documents notion of strong open score for potential future use.

DEFAULT_FALLBACK_FRAME_LABEL
- final default frame label.

These constants define the runtime aggregation policy.

----------------------------------------------------------------------
B. NORMALIZATION HELPERS
----------------------------------------------------------------------

1. _normalize_eye_state_label(label)
- normalize textual labels to canonical open/close/unknown vocabulary.

2. _get_classifier_name_from_model_bundle(model_bundle)
- read classifier_name from bundle if available.

3. _validate_model_bundle(model_bundle)
- ensure runtime bundle contains:
  - model
  - preprocessing_config
  - lbp_config
  - classifier_config

4. _convert_prediction_to_open_score(predicted_label, predicted_score, classifier_name)
- unify classifier output into one open-score in [0, 1].

Rules:
- no numeric score -> hard-label score,
- knn -> probability-like score,
- linear_svm -> sigmoid(decision value),
- otherwise -> hard-label score.

These helpers make runtime aggregation classifier-family-independent.

----------------------------------------------------------------------
C. FAILED-EYE RECORD BUILDER
----------------------------------------------------------------------

5. _build_empty_eye_prediction(eye_box, eye_index, reason)

Purpose:
- create a standard failed-eye prediction record.

Why it exists:
- later aggregation can handle failures and successes in one consistent list
  structure.

----------------------------------------------------------------------
D. SINGLE-EYE LBP CLASSIFICATION
----------------------------------------------------------------------

6. classify_single_eye_lbp(...)

Purpose:
- classify one detected eye box using the trained LBP model bundle.

Internal steps:
- validate model bundle,
- extract eye ROI using eye_state.py helper,
- if invalid -> return empty prediction record,
- prepare runtime feature record,
- run classifier inference,
- normalize class name,
- convert prediction to unified open-score,
- attach success flag and open-score,
- return structured prediction record.

This is the eye-level runtime inference function.

----------------------------------------------------------------------
E. FRAME-LEVEL AGGREGATION
----------------------------------------------------------------------

7. aggregate_eye_predictions(...)

Purpose:
- combine multiple eye-level prediction records into one frame-level label.

Policy:
- 0 valid eyes:
  use previous_eye_state if valid, otherwise fallback label.
- 1 valid eye:
  use that eye directly.
- 2+ valid eyes:
  majority hard-label vote,
  then mean open-score tie break,
  then previous_eye_state,
  then fallback label.

This function is what turns eye-level inference into frame-level inference.

----------------------------------------------------------------------
F. PUBLIC FRAME-LEVEL INTERFACE
----------------------------------------------------------------------

8. classify_eye_state_lbp(...)

Purpose:
- full public runtime frame-level classifier.

Internal branches:
- if model_bundle is missing:
  - optionally fall back directly to heuristic classifier,
  - otherwise use previous/fallback label.
- if model_bundle exists:
  - classify each eye with classify_single_eye_lbp(...),
  - aggregate with aggregate_eye_predictions(...),
  - optionally use heuristic fallback if zero valid LBP eye predictions exist.

Optional return_details:
- returns diagnostic detail structure together with the final label.

This is the main runtime function used by main.py and experiment_search.py.

======================================================================
12) MODULE: evaluation.py
======================================================================

OVERALL PURPOSE

This module is the final evaluation and reporting layer.

Its job is to take the stored frame_results from runtime processing and compare
their predicted eye-state labels against the ground-truth label file.

It computes:
- aligned sequence lengths,
- accuracy,
- confusion-style counts,
- timing summaries,
- formatted report text,
- saved evaluation report.

It is intentionally independent from how predictions were produced.

----------------------------------------------------------------------
INTERNAL STRUCTURE
----------------------------------------------------------------------

A. label normalization
B. input loading and extraction helpers
C. sequence alignment
D. accuracy and confusion-style statistics
E. timing statistics
F. full evaluation pipeline
G. formatting / printing / saving helpers

----------------------------------------------------------------------
A. LABEL NORMALIZATION
----------------------------------------------------------------------

1. normalize_eye_state_label(label)

Purpose:
- normalize textual labels into canonical evaluation vocabulary:
  - open
  - close
  - unknown / other

Why it exists:
- runtime and ground-truth sources may use slightly different label strings.

======================================================================
FINAL SUMMARY OF HOW ALL MODULES FIT TOGETHER
======================================================================

The whole project can be read as one layered system:

TOP ORCHESTRATION
- auto_select_and_run.py
    Runs search, chooses best config, launches final run.

EXPERIMENT SEARCH
- experiment_search.py
    Builds config combinations, runs experiments, ranks them.

TRAINING-DATA INGESTION
- eye_training_io.py
    Loads structured training records from the mrlEyes dataset.

SHARED IMAGE PREPROCESSING
- eye_preprocessing.py
    Standardizes eye images/ROIs into stable grayscale normalized form.

FEATURE-BRIDGE LAYER
- eye_lbp_dataset.py
    Connects records to preprocessing, LBP extraction, matrices, metadata.

LBP DESCRIPTOR ENGINE
- lbp_features.py
    Converts preprocessed eye images into numeric LBP descriptors.

SUPERVISED CLASSIFIER LAYER
- eye_lbp_classifier.py
    Trains/predicts learned models and builds reusable model bundles.

RUNTIME LBP EYE-STATE CLASSIFIER
- eye_state_lbp.py
    Uses the trained model bundle to classify eyes and aggregate frame labels.

HEURISTIC FALLBACK / LEGACY RUNTIME CLASSIFIER
- eye_state.py
    Provides original heuristic eye-state logic and eye-ROI extraction.

LOCALIZATION LAYER
- detectors.py
    Finds faces, eyes, mouth in grayscale video frames.

FINAL EXECUTION PIPELINE
- main.py
    Runs one full real video pass with one chosen configuration.

FINAL EVALUATION / REPORTING
- evaluation.py
    Compares predictions against ground truth and formats/saves final results.

So the project is not just a classifier and not just a detector. It is a full
system composed of:
- data ingestion,
- preprocessing,
- descriptor construction,
- supervised learning,
- runtime localization,
- runtime frame inference,
- experiment management,
- final orchestration,
- and evaluation/reporting.

That is the full purpose and internal structure of all the modules together.
